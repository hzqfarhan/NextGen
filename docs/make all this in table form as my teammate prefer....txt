	4. Technical Blueprint & Product Requirements Document (PRD)
4.1 Functional Requirements
Requirement ID
	Technical Module
	Functional Specification (What It Must Do)
	SR-1.1
	Streak Tracking
	Must monitor daily discretionary user card transaction inputs against the live calculated Safe Daily Spend threshold.
	SR-1.2
	Increment Logic
	At 11:59 PM, if total daily expenditure $\le$ Safe Daily Spend, the global state variable currentStreak increments by $+1$.
	SR-1.3
	Reset Logic
	If daily spend exceeds Safe Daily Spend, currentStreak drops to 0. Updates companion state to sad or angry.
	SR-1.4
	Streak Shield
	Provides an interface once a week to consume a "Streak Shield" earned via social media sharing to prevent status decay.
	SR-2.1
	Tier Computation
	Must calculate and evaluate user access tiers dynamically inside the state engine backend whenever store states update.
	SR-3.1
	Roast Export
	Must render a high-visibility canvas export action button immediately after the local dialect generation sequence completes.
	SR-3.2
	Referral Loop
	QR code targets must accept inbound parameters during onboarding (/setup), triggering a dual credit configuration hook.

Layer

	Addresses the app's No Authentication / LocalStorage Only gap. Your Ubuntu cloud instance runs a lightweight API backend (POST /api/user/sync) to persist state arrays securely into a structured relational database.

	API Key Orchestration
	Migrates the sensitive GEMINI_API_KEY entirely away from client-side vulnerability. Move the route handler proxy logic directly to your secure cloud instance utilizing server-side environment variables (process.env.GEMINI_API_KEY).
